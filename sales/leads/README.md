# Web リサーチ営業

ICHI の新規開拓を半自動化する機能です。Claude の web_search で候補店舗を探し、人間が検収した後に問い合わせフォーム用の下書きを生成します。

---

## 全体のフロー

```
criteria.md（ターゲット定義）
    ↓
npm run research  … Claude が web_search で候補を探し candidates.csv に追記
    ↓
【人間が検収】  … source_url を開いて実在・個人経営を確認、send 列に ○
    ↓
npm run draft     … send=○ の行だけ下書きを生成、drafts/ に保存
    ↓
【人間が手動投稿】 … 各下書きのフォーム URL に自分でコピペして送信
```

**位置づけ**: 「最初の1事例を作るための母数稼ぎ」ツールです。受注より無料 PoC・激安お試し（5万円〜）を取りにいきます。コールドメッセージの返信率は数 % が現実なので、1件ずつ丁寧に送ることを前提にしています。大量一斉送信はしません。

## 使い方

### 1. 候補リサーチ: `npm run research`

```bash
# 基本形（東京 下北沢 / 両業態 / 1業態あたり6件目安）
npm run research -- --area "東京 下北沢"

# 引数を全部指定する場合
npm run research -- --area "大阪 心斎橋" --type salon --count 8
```

| オプション | 説明 | 省略時 |
|-----------|------|--------|
| `--area`  | 検索する地域（**必須**） 例: `"東京 吉祥寺"` | なし（省略するとエラー） |
| `--type`  | `cafe`（カフェ・飲食店）/ `salon`（美容室・サロン・ネイルサロン）/ `both` | `both` |
| `--count` | 1業態あたりの目安件数 | `6` |

- 結果は `sales/outputs/leads/candidates.csv` に**追記**されます（既存行と source_url が重複する店はスキップ）。
- 追加ヒントを使いたい場合は `sales/inputs/leads/` に `.md` ファイルを置くと system プロンプトに添付されます。

### 2. 検収（人間が行う）

1. `candidates.csv` を Excel またはスプレッドシートで開きます（文字コード: UTF-8 BOM 付き）。
2. **各行の `source_url` を実際にブラウザで開き**、実在・個人経営・情報の一致を人間の目で確認します。
   - チェーン・FC・情報が一致しない行は `send` 列を空のままにします。
3. `contact_form_url` が空の行は、公式サイトを自分で探してフォーム URL を記入してください。メールアドレスしか連絡先がない店は送りません。
4. 送ると決めた行の `send` 列に `○` を入力します。
5. **1日あたりの件数は絞ります**（慣れるまでは 5〜10 件が目安）。

### 3. 下書き生成: `npm run draft`

```bash
npm run draft
```

- `send` 列が空でない行だけを対象に、1店1ファイルの下書きを生成します。
- 出力先: `sales/outputs/leads/drafts/`
- `contact_form_url` が未入力の店はフォーム URL 未取得として警告が表示されます（送信前に補記が必要）。

### 4. 送信（人間が行う）

生成された `drafts/` 配下の `.md` ファイルを開きます。ファイル先頭に投稿先フォームの URL が書いてあります。

1. ブラウザでそのフォーム URL を開きます。
2. 下書きの件名・本文を**手でコピー＆ペースト**します。内容を自分で読んで確認し、必要なら加筆・修正します。
3. 自分の名前・連絡先を確認して送信します。

## ディレクトリ構成

```
sales/
├── leads/
│   ├── criteria.md       … ターゲット定義・スコアリング基準（編集して使う）
│   ├── research.js       … 候補検索スクリプト
│   ├── draft.js          … 下書き生成スクリプト
│   └── README.md         … このファイル
├── inputs/
│   └── leads/            … 追加の検索ヒント（任意・*.md を置く）
└── outputs/
    └── leads/
        ├── candidates.csv        … 候補リスト（send 列を人間が編集）
        ├── _last_research_raw.txt … パース失敗時の生レスポンス（デバッグ用）
        └── drafts/               … 店ごとの下書き（*.md）
```

## ターゲット定義の変更

`criteria.md` を編集すれば、ターゲット業態・スコアリング基準・訴求軸をカスタマイズできます。次回 `npm run research` 実行時から新しい定義が使われます。

## 安全・コンプライアンス（重要・必ず守る）

> **送信（フォーム投稿）は必ず人間が手動で行うこと。自動投稿は実装されていないし、してはいけません。**

| ルール | 内容 |
|--------|------|
| 送信は手動 | フォームへの投稿は人間が1件ずつ手でコピペして行う。スクリプトは下書き保存までしか行わない |
| スクレイピング禁止 | サイトの自動巡回・データ抜き取りは行わない。Claude の web_search で得た公開情報のみを使用する |
| メールアドレス禁止 | 個人のメールアドレスは収集・保存・送信先にしない。記録するのは公開された問い合わせフォーム URL のみ |
| 出典 URL 必須 | すべての店舗情報に出典 URL が必須。出典 URL のない店は出力されない。**送信前に必ず出典 URL を自分で開いて実在を確認する** |
| 法令遵守 | 特定電子メール法・個人情報保護法・各サイト利用規約に沿った運用を心がける。判断に迷う送信先は送らない |

## トラブルシューティング

| 症状 | 確認ポイント |
|------|-------------|
| `ANTHROPIC_API_KEY が読み込めません` | リポジトリ直下の `.env` に `ANTHROPIC_API_KEY=sk-...` が設定されているか確認 |
| `JSON のパースに失敗しました` | `outputs/leads/_last_research_raw.txt` を開いて Claude の出力を確認。`--count` を減らして再実行 |
| API 429 エラー（レート制限） | research と draft の間に 1〜2 分空けて再実行。draft.js はリトライを自動で行う |
| 生成件数が少ない | `--count` を増やすか、`--area` を広めに指定（例: `"東京 吉祥寺周辺"`）して再実行 |
| 下書きが長い・短い | 現状は Claude に依存。`draft.js` の system プロンプトの字数指定を調整可能 |

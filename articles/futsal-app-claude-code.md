## はじめに

フットサルの個人参加イベント（個サル）を主催していると、参加者管理がじわじわと面倒になってくる。最初はExcelで管理していたが、キャンセル待ちが出た瞬間に手作業の限界を感じた。「誰が何番目のキャンセル待ちか」を毎回手で並べ替え、LINEで連絡して、Excelを更新して——その繰り返しに1時間以上かかることもあった。「これはアプリにするべきだ」と思い立ち、Claude Codeを使って実際に作ってみた。

---

## 作ったもの

**フットサル個サル参加者管理Webアプリ**（GitHub Pages上で動作）

参加者がフォームから申し込み→定員内なら確定、超えたらキャンセル待ち列に自動登録→主催者がブラウザで一覧確認・承認・キャンセル処理できる、という一連の流れをカバーする。

**主な機能:**

- **参加者登録フォーム** — 氏名・連絡先・参加希望日を入力して送信
- **キャンセル待ち管理** — 定員超過時に自動で待機列に追加、繰り上がりも自動処理
- **主催者管理画面** — Google OAuthでログインした主催者のみアクセス可能、参加者一覧の表示・承認・キャンセル操作
- **CSV出力** — 当日の参加者リストをワンクリックでダウンロード
- **Google OAuth 2.0認証** — 主催者アカウントのみ管理画面にアクセスできるよう制限
- **Google Sheetsをデータベース代わりに使用** — サーバーレスで運用コスト¥0を実現

## Claude Codeへの指示方法

### 要件の渡し方

最初に「全部まとめて作って」と指示するのではなく、機能単位で分割して渡した。Claude Codeは長い要件を一度に処理しても精度が落ちることがあるため、次のように段階的に指示した。

**プロンプト例 1: 基本フォームの生成**

```
以下の仕様でHTML/CSS/JSのフォームを作成してください。

- 入力項目: 氏名（必須）、メールアドレス（必須）、電話番号（任意）
- バリデーション: メール形式チェック、必須項目の未入力チェック
- 送信後: 確認メッセージを表示（ページ遷移なし）
- スタイル: シンプルなモノトーン、モバイルファースト
- 外部ライブラリは使用しない
```

**プロンプト例 2: Google Sheets書き込み処理**

```
Google Sheets APIを使って、フォームの送信内容をスプレッドシートに追記するJavaScript関数を書いてください。

- スプレッドシートID: [SPREADSHEET_ID]
- 書き込み先シート: "参加者リスト"
- 書き込む列: A列=タイムスタンプ、B列=氏名、C列=メール、D列=ステータス（"確定" or "キャンセル待ち"）
- アクセストークンは引数で受け取る形にする
- fetch APIを使用すること
```

**プロンプト例 3: キャンセル待ちロジック**

```
以下の仕様でキャンセル待ち管理関数を実装してください。

- Google Sheetsから現在の参加者数を取得する
- 定員（const MAX_PARTICIPANTS = 20）を超えていればステータスを"キャンセル待ち"にする
- キャンセル待ちの場合、待機番号（何番目か）を返す
- 定員以内であれば"確定"ステータスで登録する
```

### そのまま使えた部分・修正が必要だった部分

**そのまま使えた部分:**

- フォームのHTML/CSSとバリデーションロジック
- Google Sheets APIへのfetch処理（appendValuesのエンドポイント指定）
- CSVエクスポート関数（Blob生成〜ダウンロードリンク生成まで）

**修正が必要だった部分:**

- Google OAuthのコールバック処理。Claudeが生成したコードは`redirect_uri`に`localhost`を使っており、GitHub Pagesのhttps環境では動作しなかった。自分でリダイレクトURIをGitHub PagesのURLに変更した。
- スプレッドシートへの書き込み時、列の順番がずれる問題。Claudeのコードは`values: [[col1, col2, col3]]`の順序を指定の逆にしていたため、自分でデバッグして修正した。

## ハマったポイントと解決策

### 1. Google OAuthがローカル（file://）で動作しない

最初はHTMLファイルをブラウザで直接開いて（`file://`）開発していた。フォームやCSS確認は問題なかったが、Google OAuthのログインボタンを押した瞬間に以下のエラーが出た。

```
Error 400: redirect_uri_mismatch
```

Google OAuth 2.0は、**リダイレクトURIとして`https://`または`http://localhost`しか受け付けない**。`file://`プロトコルは許可されていないため、ローカルファイルから直接OAuth認証を呼び出すことはできない。

**解決策:** 早めにGitHub Pagesへデプロイしてしまう。

```bash
# リポジトリをGitHub Pagesで公開
# Settings > Pages > Source: main branch / root

# Google Cloud ConsoleでリダイレクトURIを追加
# https://[username].github.io/[repo-name]/
```

Google Cloud Consoleの「認証情報」→「OAuthクライアントID」の設定で、承認済みリダイレクトURIに `https://[username].github.io/[repo-name]/` を追加すれば動作する。開発中は毎回GitHubにpushして確認する手間はあるが、それよりローカルサーバーを立てるほうが面倒だったためこの方法を選んだ。

### 2. Google Sheets APIの認証スコープ設定でハマった

Google Sheets APIを呼び出す際、最初は読み取りしかできなかった。書き込み（append）を実行すると`403 Forbidden`が返ってきた。

原因はOAuthリクエスト時のスコープ設定が読み取り専用になっていたためだ。

```javascript
// NG: 読み取りのみ
const SCOPES = 'https://www.googleapis.com/auth/spreadsheets.readonly';

// OK: 読み書き両方
const SCOPES = 'https://www.googleapis.com/auth/spreadsheets';
```

Claude Codeに「Google Sheets APIで読み書きするためのOAuthスコープ設定を教えて」と聞いたところ、上記の違いを即座に説明してくれた。あわせて「スコープを変更した場合はユーザーに再認証させる必要がある」という注意点も教えてもらえた。

スコープを `spreadsheets`（読み書き）に変更し、一度ログアウトして再ログインしたところ、書き込みが正常に動作した。

## 完成して気づいたこと

**要件定義→実装のサイクルが異常に速い。**

通常、「フォームを作る→Sheets APIを繋ぐ→認証を実装する」という工程は、APIドキュメントを読んだり、サンプルコードを探したりするだけで数時間かかる。Claude Codeに渡せば、土台となるコードが数分で出てくる。自分がやることは「動かして確認する」「要件と違う部分を修正する」だけだった。

結果として3日間で実用レベルのWebアプリが完成した。内訳は、1日目が設計とフォーム実装、2日目がSheets API連携と認証、3日目がキャンセル待ちロジックと管理画面の仕上げだ。

このプロセスで気づいたのは、「動くものを作る」ことに集中できるという点だ。BoilerplateコードやAPIの定型処理をゼロから書く必要がなく、ロジックと要件の整合性に集中できる。

このフットサルアプリの開発体験が、「Claude Codeを使えば一人でも複数の事業を動かせる」という確信につながった。その後、受託開発・コンテンツ生成・バックオフィス自動化など各部門を同じアプローチで構築していくICHIの起点になっている。
